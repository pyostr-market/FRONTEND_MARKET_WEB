Цель этого документа — описать архитектуру фронтенд-приложения Marketplace на React. Документ предназначен для использования ИИ-агентом и разработчиками как источник правил и принципов, которые необходимо соблюдать при разработке новых модулей, страниц и компонентов.

Общие принципы

Приложение является большим маркетплейсом с большим количеством данных и интеграций с множеством backend сервисов. Backend реализован на Python и предоставляет данные через HTTPS API. Архитектура фронтенда должна обеспечивать масштабируемость, разделение ответственности, предсказуемость структуры проекта и удобство для автоматической генерации и анализа кода.

Основные принципы:

1. Все приложение должно быть разделено на слои.
2. Каждый слой имеет строго определенную ответственность.
3. Компоненты должны быть максимально изолированы.
4. Бизнес-логика не должна находиться внутри UI компонентов.
5. Вся навигация должна использовать централизованную систему маршрутов.
6. Все API вызовы должны проходить через единый слой API.
7. Повторно используемые элементы должны находиться в shared слое.
8. В коде запрещены "магические строки" для маршрутов и API путей.
9. Состояние должно быть разделено на серверное и клиентское.
10. Архитектура должна быть ориентирована на масштабирование.

Основная структура проекта

Структура проекта разделена на несколько основных слоев:

app
pages
widgets
features
entities
shared

Каждый слой имеет собственную зону ответственности.

Слой app

Слой app отвечает за инициализацию приложения. Здесь не должно находиться бизнес-логики.

В слое app находятся:

инициализация роутера
глобальные провайдеры
инициализация состояния
инициализация API клиента
глобальные конфигурации

Пример содержимого слоя:

app/router
app/providers
app/store
app/config

В файле App происходит подключение:

BrowserRouter
AppProviders
AppRouter

AppProviders содержит все глобальные провайдеры приложения, например:

QueryClientProvider
AuthProvider
StateProvider
ThemeProvider

Слой pages

Pages — это страницы приложения. Каждая страница представляет собой композицию виджетов.

Страницы не должны содержать сложной бизнес-логики. Их задача — собирать UI из widgets и features.

Примеры страниц:

HomePage
CatalogPage
ProductPage
CartPage
ProfilePage
SearchPage

Каждая страница должна находиться в собственной папке.

Слой widgets

Widgets — это крупные блоки интерфейса, из которых строятся страницы.

Widgets могут содержать:

несколько компонентов
локальную логику
подключение features

Примеры widgets:

Header
Footer
ProductGrid
FiltersPanel
PromoCarousel
CategoriesGrid
BrandsCarousel
BestSellers

Widgets могут использовать entities и features.

Слой features

Features описывает действия пользователя.

Feature представляет собой отдельную бизнес-операцию.

Примеры features:

add-to-cart
add-to-favorites
compare-products
login
logout
search-products
apply-filters

Feature может содержать:

UI компонент
бизнес-логику
работу с API
локальное состояние

Features могут использовать entities.

Слой entities

Entities описывают бизнес-сущности приложения.

Примеры сущностей:

product
category
user
cart
order
brand

Каждая сущность содержит:

тип данных
API методы
UI элементы
селекторы

Примеры UI компонентов внутри product:

ProductCard
ProductGallery
ProductPrice
ProductBadge
ProductRating

Entities не должны знать о features.

Слой shared

Shared содержит инфраструктуру и переиспользуемые элементы.

Примеры содержимого:

shared/api
shared/ui
shared/hooks
shared/utils
shared/constants
shared/config

В shared/ui находятся универсальные UI компоненты:

Button
Input
Modal
Spinner
Dropdown
Tabs

Эти компоненты не должны содержать бизнес-логики.

Навигация и маршруты

В приложении используется централизованная система маршрутов.

Все маршруты определяются в конфигурационном файле.

Файл paths содержит список всех URL путей.

Примеры путей:

home
catalog
category
product
cart
profile
orders
login

Второй файл routes содержит конфигурацию роутера.

Каждый маршрут содержит:

path
component
public или private
layout

Private маршруты доступны только авторизованным пользователям.

Для защиты маршрутов используется компонент PrivateRoute.

Навигация внутри приложения должна использовать:

useNavigate
Link

Все переходы должны использовать значения из paths или routeBuilder.

Использование строковых URL напрямую запрещено.

Динамические маршруты

Для динамических URL используются генераторы путей.

Примеры:

product(id)
category(slug)
brand(slug)

Это позволяет централизованно управлять структурой URL.

Layouts

В приложении используется система layouts.

Layouts определяют структуру страницы.

Основные layouts:

MainLayout
ProfileLayout
AdminLayout

MainLayout используется для большинства страниц и содержит:

Header
Footer
контент страницы

ProfileLayout используется для страниц пользователя и содержит:

Header
Sidebar
контент профиля

API слой

Все взаимодействие с backend происходит через слой API.

API разделен по сервисам backend.

Примеры сервисов:

catalogApi
userApi
orderApi
searchApi
crmApi

Все API методы должны находиться внутри соответствующих модулей.

UI компоненты не должны выполнять прямые HTTP запросы.

Для работы с серверным состоянием используется React Query.

Состояние приложения

Состояние разделяется на два типа:

серверное состояние
клиентское состояние

Серверное состояние:

данные каталога
данные товаров
поисковые результаты
категории
бренды

Для серверного состояния используется React Query.

Клиентское состояние:

корзина
избранное
сравнение
пользовательская сессия
UI состояние

Для клиентского состояния используется глобальный store.

Фильтры каталога

Фильтры каталога должны храниться в URL параметрах.

Пример:

/catalog/smartphones?brand=apple&memory=256&priceFrom=500

Это позволяет:

делиться ссылками
поддерживать SEO
сохранять состояние фильтров

Компоненты

Компоненты должны быть максимально простыми.

UI компоненты должны:

принимать данные через props
не содержать бизнес-логики
не делать API запросы

Бизнес-логика должна находиться:

в hooks
в features
в entities

Hooks

Логика компонентов должна выноситься в hooks.

Примеры:

useProduct
useCatalog
useFilters
useSearch

Hooks могут обращаться к API и состоянию.

Работа с изображениями

Изображения товаров должны загружаться лениво.

Изображения должны использовать оптимизированные форматы.

Списки товаров должны поддерживать виртуализацию при больших объемах данных.

Производительность

Страницы должны загружаться лениво.

Код должен быть разделен по страницам.

Большие списки должны использовать виртуализацию.

API запросы должны кешироваться.

Общие правила

Запрещено:

использовать строковые URL напрямую
вызывать API из UI компонентов
хранить бизнес-логику внутри компонентов
дублировать типы данных

Рекомендуется:

использовать централизованные типы
использовать hooks для логики
разделять код по слоям
переиспользовать shared компоненты

Цель архитектуры

Данная архитектура предназначена для создания масштабируемого маркетплейса с большим количеством товаров, пользователей и интеграций. Архитектура должна позволять легко добавлять новые страницы, новые функции и новые сервисы backend без необходимости переписывать существующий код.
